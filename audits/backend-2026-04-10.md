## CRITICAL

### C-01: Webhook-Endpoints ohne Authentifizierung

**File:** `backend/src/routes/webhook.routes.ts:9-26`  
**Problem:** Beide Webhook-Endpoints (`/ranking-auto-accept`, `/training-reminder`) sind komplett ungeschuetzt. Kein Auth-Header, kein Secret, kein Rate-Limiting. Jeder kann per POST Ranglisten-Challenges auto-akzeptieren oder Push-Benachrichtigungen an alle User triggern.  
**Risiko:** Manipulation der Rangliste, Spam-Benachrichtigungen, DoS  
**Fix:**
```typescript
// middleware/webhookAuth.ts
export function requireWebhookSecret(req: Request, res: Response, next: NextFunction): void {
  const secret = req.headers['x-webhook-secret'];
  if (secret !== env.N8N_WEBHOOK_SECRET) {
    res.status(401).json({ success: false, error: { code: 'UNAUTHORIZED', message: 'Invalid webhook secret' } });
    return;
  }
  next();
}

// webhook.routes.ts
router.post('/ranking-auto-accept', requireWebhookSecret, async (req, res, next) => { ... });
router.post('/training-reminder', requireWebhookSecret, async (req, res, next) => { ... });
```

### C-02: Login ohne clubId-Filter (Multi-Tenant-Breach)

**File:** `backend/src/services/auth.service.ts:163-166`  
**Problem:** `login()` sucht User nur nach `email` + `isActive`, ohne `clubId`. Bei gleichem E-Mail in mehreren Clubs wird der erste Treffer zurueckgegeben — der User koennte im falschen Club landen und Zugriff auf fremde Daten bekommen.
```typescript
const dbUser = await prisma.user.findFirst({
  where: { email, isActive: true },  // ← kein clubId!
  include: { roles: true },
});
```
**Risiko:** Cross-Tenant-Datenzugriff, Privilege Escalation  
**Fix:** `clubCode` im Login-Request mitsenden, dann Club auflösen und `clubId` in die Query:
```typescript
export async function login(email: string, password: string, clubCode: string) {
  const club = await prisma.club.findUnique({ where: { clubCode } });
  if (!club) throw new AuthError('Ungueltiger Club', 'INVALID_CLUB', 400);

  const dbUser = await prisma.user.findUnique({
    where: { email_clubId: { email, clubId: club.id } },
    include: { roles: true },
  });
  // ...
}
```

### C-03: Push-Endpoints ohne Input-Validation

**File:** `backend/src/routes/push.routes.ts:11-14`  
**Problem:** POST `/` nimmt `token` und `platform` direkt aus `req.body` ohne Zod-Validation. Beliebige Werte werden an den Service weitergereicht.
```typescript
const { token, platform } = req.body;  // ← kein validate()
```
**Risiko:** Ungueltige Daten in DB, potenzielle Injection  
**Fix:**
```typescript
// shared: push.schema.ts
export const registerPushTokenSchema = z.object({
  token: z.string().min(1),
  platform: z.enum(['IOS', 'ANDROID']),
});

// push.routes.ts
router.post('/', validate(registerPushTokenSchema), async (req, res, next) => { ... });
```

## HIGH

### H-01: N+1 Query — Team Channel Sync

**File:** `backend/src/services/team.service.ts:135-142`  
**Problem:** `addTeamMember()` holt alle Team-Channels, dann upserted in einer Schleife pro Channel:
```typescript
const channels = await prisma.channel.findMany({ where: { teamId }, select: { id: true } });
for (const ch of channels) {
  await prisma.channelMember.upsert({ ... });  // N Queries!
}
```
Gleiches Muster bei `removeTeamMember()` (Zeile 156-159).  
**Risiko:** Bei 10 Team-Channels = 10+ DB-Queries pro Member-Add/Remove  
**Fix:**
```typescript
// addTeamMember: Batch-Upsert via createMany + skipDuplicates
const channels = await prisma.channel.findMany({ where: { teamId }, select: { id: true } });
await prisma.channelMember.createMany({
  data: channels.map(ch => ({ channelId: ch.id, userId })),
  skipDuplicates: true,
});

// removeTeamMember: Batch-Delete via IN
const channelIds = channels.map(ch => ch.id);
await prisma.channelMember.deleteMany({
  where: { channelId: { in: channelIds }, userId },
});
```

### H-02: N+1 Query — autoAcceptExpired Loop

**File:** `backend/src/services/ranking.service.ts:268-275`  
**Problem:** Expired Challenges werden einzeln in einer Schleife geupdated:
```typescript
for (const challenge of expired) {
  const updated = await prisma.rankingChallenge.update({
    where: { id: challenge.id },
    data: { status: 'ACCEPTED' },
  });
  results.push(updated);
}
```
**Risiko:** Bei 100 abgelaufenen Challenges = 100 DB-Queries (wird per n8n-Webhook getriggert)  
**Fix:**
```typescript
const result = await prisma.rankingChallenge.updateMany({
  where: { status: 'PENDING', deadline: { lte: now } },
  data: { status: 'ACCEPTED' },
});
return { accepted: result.count };
```

### H-04: Fehlende Transaction bei Rang-Tausch

**File:** `backend/src/services/ranking.service.ts:63-97`  
**Problem:** `updateRankingFromResult()` fuehrt 5 separate DB-Updates durch (wins, losses, 3x rank-swap). Bei Fehler mitten im Swap koennen beide Spieler `rank: 0` haben:
```typescript
await prisma.ranking.update({ data: { rank: 0 } });         // Step 1

await prisma.ranking.update({ data: { rank: loserRank } });  // Step 3 ← Fehler hier?
```

**Risiko:** Inkonsistente Rangliste, unique-constraint violation

### H-05: Fehlende Rate-Limits auf /refresh und /logout

**File:** `backend/src/routes/auth.routes.ts:53-66, 68-79`  
**Problem:** `/register` und `/login` haben `authLimiter` (5 Req/15min), aber `/refresh` und `/logout` sind unlimitiert. Ein Angreifer mit gueltigem Refresh-Token kann den Server mit Requests fluten.
```typescript
router.post('/refresh', validate(refreshTokenSchema), async (...) => { ... });  // ← kein authLimiter
router.post('/logout', async (...) => { ... });                                  // ← kein authLimiter
```
**Risiko:** Resource Exhaustion, Token-Brute-Force  
**Fix:** `authLimiter` auf beide Endpoints anwenden.

### M-09: Soft-Delete vs Hard-Delete Inkonsistenz

**Problem:** Inkonsistente Loeschstrategie ueber das gesamte Schema:

- **Hard-Delete:** Team (team.service.ts:113), Event (event.service.ts:143), Message (channel.service.ts:220), TeamMember, Availability

**Risiko:** Geloeschte Teams/Events sind unwiderruflich weg. Kein Audit-Trail fuer geloeschte Daten.

### M-10: Logger ohne Sensitive-Data-Redaction

**File:** `backend/src/utils/logger.ts:14-31`  
**Problem:** `formatMessage()` serialisiert den gesamten `meta`-Parameter als JSON, ohne sensitive Felder zu filtern. Wenn ein Entwickler versehentlich `{ password, token }` loggt, landet das im Output.
```typescript
const metaStr = filteredMeta ? ` ${JSON.stringify(filteredMeta)}` : '';
// ← kein Redaction von password, token, secret, etc.
```
**Risiko:** Credential-Leak in Log-Files  
**Fix:**
```typescript
const REDACT_KEYS = ['password', 'token', 'secret', 'authorization', 'cookie'];
function redactSensitive(meta: Record<string, unknown>): Record<string, unknown> {
  return Object.fromEntries(
    Object.entries(meta).map(([k, v]) => [
      k,
      REDACT_KEYS.some(r => k.toLowerCase().includes(r)) ? '[REDACTED]' : v,
    ])
  );
}
```
